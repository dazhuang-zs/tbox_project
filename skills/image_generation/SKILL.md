## ⚠️ 执行规范：必须严格按步骤生成图片

> **每次生成图片，必须严格按照以下步骤顺序执行，不得跳过、合并或调换任何步骤。**

```
步骤1：提取尺寸参数         → 严格参考【核心模块 / 尺寸参数提取】
步骤2：提取提示词           → 严格参考【核心模块 / 提示词智能提取】
步骤3：调用 MCP 工具生成图片，并输出图片链接
  ├─ 调用方式：严格参考 references/mcp_service.md
  ├─ ⚠️ 调用工具1时 --timeout 300000 为必填，缺少则视为无效调用
  ├─ 链接输出：严格参考 references/image_link_constraints.md
  └─ 所有工具均失败时 → 输出"图片生成失败了，请稍后再试。"，终止
```

> ⛔ **严禁以下行为**：
> - 禁止使用 curl、fetch、http 请求或任何其他方式直接访问 MCP 端点
> - 禁止跳过步骤1/2，直接构造参数调用 MCP
> - 必须完整走完步骤1→2→3，不得自行简化流程
> - **调用命令**：通过 `mcporter call` 命令调用 MCP 工具；**只能使用 `key=value` 显式传参**，`--args '{...}'` 和 `--json` 均严禁使用
> - **tool selector 必须用点号**：`multimodal-model.generate_image_modelX_with_watermark`，严禁空格分隔
> - **工具1超时必须显式设置**：调用工具1前必须先执行 `export MCPORTER_CALL_TIMEOUT=300000`，并在调用时传入 `--timeout 300000`，两者缺一不可；工具2无需设置
> - **需求简单并非省略步骤的理由**：无论用户需求多简单，步骤1和步骤2都必须完整执行，不得以"需求简单"为由跳过

## 服务架构

本技能基于 **multimodal-model** MCP 服务实现图像生成功能：

### MCP 工具

提供两个图像生成工具，根据图片尺寸自动选择：

1. **generate_image_model1_with_watermark** (优先调用)
   - 参数：prompt, aspectRatio, imageSize, referenceImageUrls

2. **generate_image_model2_with_watermark** (降级调用)
   - 参数：prompt, size, referenceImageUrls

### 支持功能

- **文生图**：根据文字描述生成图片
- **图生图**：基于参考图生成新图片
  - 图片编辑：修改、裁剪、删除元素等
  - 图片融合：多张图片融合、合成
  - 风格迁移：基于参考图进行风格转换

### 重要特性

- **智能工具选择**：根据图片宽高比自动选择最优工具
- **降级机制**：主工具调用失败时自动降级到备用工具
- **多轮对话**：支持根据对话上下文自动识别和使用上一轮生成的图片作为参考图
- **参考图来源**：支持用户上传图片和上一轮生成的图片两种来源
- **图片输出格式**：所有生成的图片必须**同时**输出两种格式（缺一不可、两行紧邻、URL 完全相同），确保 web 端和 IM 渠道（微信、飞书等）都能正常展示：
  - `![图片描述](图片链接)` — webchat 内联渲染图片
  - `MEDIA:图片链接` — IM 渠道由运行时提取 MEDIA: 令牌后投递为原生图片消息（微信、飞书等）
- **图片链接约束**：详见[图片链接输出约束](#图片链接输出约束)章节

## 核心模块

### 🔧 尺寸参数提取

图片参数解析专家，擅长从用户需求中准确提取图片尺寸、分辨率等关键信息。

#### 提取的尺寸参数

| 参数名     | 类型   | 说明                                         | 示例值           |
| ---------- | ------ | -------------------------------------------- | ---------------- |
| radio      | string | 处理后的有效尺寸，必须在 [1024, 4096] 区间内 | "1366x1024"      |
| image_size | string | 图片分辨率                                   | "1K", "2K", "4K" |

#### 核心处理逻辑

##### 任务 1: 获取尺寸 size

- **分析用户需求**：检查是否包含图片比例或宽高信息
- **默认处理**：如果没有明确说明 → `size = ''`（无明确说明时保持空值）
- **明确说明时**：
  - 比例换算为具体宽高值（如 "16:9" → "1920×1080"）
  - 格式必须为 `"宽度x高度"`
  - **特殊处理**：宫格类图片取最终合成图的比例或宽高信息

##### 任务 2: 获取有效尺寸 radio（自动缩放机制）

确保尺寸在模型支持范围 [1024, 4096] 内，自动执行同比例缩放：

**缩放规则**：
```
- 如果 W < 1024 或 H < 1024：scale = 1024 / min(W, H)
- 如果 W > 4096 或 H > 4096：scale = 4096 / max(W, H)
- 新尺寸：NW = round(W × scale), NH = round(H × scale)
- 循环检查直至 NW 和 NH 都在 [1024, 4096] 区间内
```

**缩放示例**：

示例 1（尺寸过小，需要放大）：
```
原始尺寸：800x600
问题：H(600) < 1024
计算：scale = 1024/600 ≈ 1.707
结果：NW = round(800×1.707) = 1366, NH = round(600×1.707) = 1024
输出：radio = "1366x1024"
```

示例 2（尺寸过大，需要缩小）：
```
原始尺寸：5000x3000
问题：W(5000) > 4096
计算：scale = 4096/5000 = 0.8192
结果：NW = round(5000×0.8192) = 4096, NH = round(3000×0.8192) = 2458
输出：radio = "4096x2458"
```

##### 任务 3: 获取图片分辨率 image_size

- **识别**：用户需求中的分辨率描述："1K", "2K", "4K"
- **默认值**：如果没有说明或值为 "3K" → `image_size = "2K"`
- **有效值**：只能是 `"1K"`, `"2K"`, `"4K"`

#### 输出格式

严格输出 JSON 格式的 params 对象：
```json
{
  "radio": "2048x2048",
  "image_size": "2K"
}
```

#### 关键限制

- `radio` 必须严格满足格式 `"宽度x高度"`，且宽高值都在 [1024, 4096] 区间内
- `image_size` 只能是 `"1K"`, `"2K"`, `"4K"` 三种取值
- 不允许随意补充用户未明确指定的参数

### 🎨 提示词智能提取

提示词处理专家，能够根据用户需求智能识别意图类型，并生成遵循用户意图的中文提示词。

#### 提示词类型识别

系统会自动识别以下四种意图类型：

| 类型代码 | 意图类型      | 说明                        | 触发条件                                                     |
| -------- | ------------- | --------------------------- | ------------------------------------------------------------ |
| 1        | 宫格/网格图片 | 生成多张图片合成宫格        | "三宫格"、"四宫格"、"3x3网格"等                              |
| 2        | 差异化多图    | 生成多张不同风格/场景的图片 | 要求不同视角、角度、场景、风格等                             |
| 3        | 图生图        | 基于参考图生成/编辑/融合    | 包含"修改"、"裁剪"、"融合"、"合成"等关键词**或用户上传图片** |
| 4        | 常规生图      | 标准的单张或多张图片生成    | 不满足以上特殊条件的常规需求，**且用户未上传图片**           |

⚠️ **类型识别关键原则**：
- **用户上传图片 = 类型3**：无论用户描述如何，只要用户上传了图片，就必须识别为类型3（图生图）
- **优先级顺序**：类型3 > 类型1 > 类型2 > 类型4

#### 意图识别流程

```
判断逻辑（按优先级顺序）：
步骤1：检查用户是否上传了图片
  └─ 如果用户上传了图片 → 输出类型 3（图生图）⚠️ 最高优先级

步骤2：检测图生图关键词
  └─ 如果包含图生图关键词 → 输出类型 3（图生图）

步骤3：检测宫格关键词
  └─ 如果包含宫格/网格描述 → 输出类型 1（宫格/网格图片）

步骤4：检测差异化关键词
  └─ 如果包含差异化需求 → 输出类型 2（差异化多图）

步骤5：其他情况
  └─ 输出类型 4（常规生图）
```

> 📖 **详细说明**：各类型的详细触发条件、处理特点、示例和注意事项，严格参考 [references/prompt_types.md](references/prompt_types.md)

#### 提示词生成规则

```
严格禁止（适用于所有类型，无例外）：
❌ 禁止修改用户的描述内容（不能改写、换词、调整语序、同义替换）
❌ 禁止精简用户的描述内容（不能删减、缩写、省略任何细节）
❌ 禁止截断用户的描述内容（不能只取部分、丢弃后半段、遗漏中间内容）
❌ 禁止总结用户的描述内容（不能用概括性语言替代原文、不能提炼要点）
❌ 禁止翻译用户的描述内容（用户用中文就保持中文，用英文就保持英文）

处理规则：
1. 完整保留用户的原始描述内容
   - 人名、地名、品牌名等专有名词必须原样保留
   - 用户描述的顺序不得调整
2. 仅移除以下非描述性信息（这是唯一允许的删减操作）：
   - 分辨率、尺寸单位信息（类型1保留比例信息）
   - 图片文件信息（如 xxx.jpg/xxx.png）
   - 数量信息（如"生成3张"中的"3张"）
```

#### 数量识别

```
识别规则：
- 默认值：num = 1
- "生成X张图片"、"X种风格"、"X个场景" → num = X
- "基于已生成X个场景/风格/图片做修改" → num = X
- 排除文档结构序号（如"1. 功能更新"不计入）
- 排除引用标记（如"参考图1、图2"不计入）
```

## MCP 服务调用

### 前置步骤

在调用MCP工具之前，**必须优先执行 `get_size.py` 脚本**获取模型需要的尺寸配置信息。

```bash
python3 skills/image_generation/scripts/get_size.py '{"radio":"2048x2048","image_size":"2K"}'
# 输出: {"use_tool1": true, "image_size": "2K", "image_config": {"aspect_ratio": "1:1", "image_size": "2K"}}
```

### 工具选择逻辑

根据 `get_image_size` 返回的 `use_tool1` 字段决定调用哪个工具：

| use_tool1值 | 优先调用工具 | 降级调用工具 |
| ------------ | ------------ | ------------ |
| `True`       | 工具1        | 工具2        |
| `False`      | 工具2        | 无           |

### 调用示例

**工具1（use_tool1=True 时优先调用）**：

```bash
export MCPORTER_CALL_TIMEOUT=300000
mcporter call multimodal-model.generate_image_model1_with_watermark \
  --timeout 300000 \
  prompt="一只可爱的猫咪在阳光下睡觉" \
  aspectRatio="1:1" \
  imageSize="2K"
```

⚠️ **工具1仅用于文生图**，不支持图生图（类型3）。

**工具2（use_tool1=False、工具1失败、或图生图时调用）**：

```bash
mcporter call multimodal-model.generate_image_model2_with_watermark \
  prompt="一只可爱的猫咪在阳光下睡觉" \
  size="2048x2048"
```

`size` 取 `image_size` 字段；工具2无需设置超时。

## 图片链接输出约束

### ⚠️ 双格式强制输出（缺一不可）

每张生成的图片必须**同时**输出以下两行（紧邻、各占一行、URL 完全相同），覆盖 web 端和 IM 渠道：

```
![图片描述](图片链接)
MEDIA:图片链接
```

- **第一行 `![](url)`**：webchat 内联渲染为图片
- **第二行 `MEDIA:url`**：运行时从模型输出中提取 `MEDIA:` 令牌，将 URL/path 转交给 IM 渠道的原生图片投递（微信会将其作为原生图片消息发送；飞书会下载图片再以 `msg_type=image` 发送）

**为什么需要两种格式？**

- 微信渠道：插件的 markdown-filter 会**剥除** `![](url)`，仅 markdown 时图片完全不显示。必须有 `MEDIA:` 行才能展示
- 飞书渠道：text 消息不渲染 markdown 图片语法，必须靠 `MEDIA:` 行投递原生图片
- webchat：渲染 markdown，靠 `![](url)` 行内联展示；运行时会从 text 中剥除 `MEDIA:` 行，不会有残留文本

**严禁的输出形式：**

- ❌ 仅输出 `![](url)`：微信、飞书都看不到图片
- ❌ 仅输出 `MEDIA:url`：webchat 体验降级（无内联图）
- ❌ 两行 URL 不一致：微信 / 飞书显示的图片和 webchat 显示的不是同一张

### 核心原则（URL 字符保真度）

**MCP 返回什么链接，就原样输出什么链接！**

- **零修改**：不修改任何字符
- **零添加**：不添加任何字符
- **零删除**：不删除任何字符
- **零优化**：不优化任何字符
- **零缓存**：严格禁止使用缓存的图片链接，必须每次都使用 skill 实时生成的图片链接

### 关键提醒

1. **不要凭记忆输入链接**：必须从系统日志中复制
2. **不要优化链接格式**：MCP 返回什么就输出什么
3. **不要修改任何字符**：包括大小写、特殊字符、连续相同字符
4. **必须逐字符核对**：确保输出链接与系统返回链接完全一致
5. **连续相同字符需精确复制**：如 MCP 返回 `AAAA`（4个A），不能输出 `AAA`、`AAAAA` 或更多/更少个A
6. **严禁使用缓存链接**：必须每次都使用 skill 实时生成的图片链接
