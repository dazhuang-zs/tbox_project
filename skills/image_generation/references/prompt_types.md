# 提示词类型详细说明

本文档详细说明四种提示词类型的识别规则和处理方式。

---

## 类型识别概览

| 类型代码 | 意图类型      | 说明                        | 触发条件                                                     |
| -------- | ------------- | --------------------------- | ------------------------------------------------------------ |
| 1        | 宫格/网格图片 | 生成多张图片合成宫格        | "三宫格"、"四宫格"、"3x3网格"等                              |
| 2        | 差异化多图    | 生成多张不同风格/场景的图片 | 要求不同视角、角度、场景、风格等                             |
| 3        | 图生图        | 基于参考图生成/编辑/融合    | 包含"修改"、"裁剪"、"融合"、"合成"等关键词**或用户上传图片** |
| 4        | 常规生图      | 标准的单张或多张图片生成    | 不满足以上特殊条件的常规需求，**且用户未上传图片**           |

⚠️ **类型识别关键原则**：详见[核心处理流程 - 意图识别](#意图识别流程)

---

## 意图识别流程

```
关键词检测：
- 图生图关键词：切割、分割、裁剪、修改、处理、编辑、去除、删除、融合、合成、混合、叠加
- 宫格关键词：X宫格、XxX网格、X×X网格
- 差异化关键词：不同、多种、各种、多张、多个
- 用户上传图片：检查用户消息中是否包含图片附件

判断逻辑（按优先级顺序）：
步骤1：检查用户是否上传了图片
  └─ 如果用户上传了图片 → 输出类型 3（图生图）⚠️ 最高优先级

步骤2：检测图生图关键词
  └─ 如果包含图生图关键词 → 输出类型 3（图生图）

步骤3：检测宫格关键词
  └─ 如果包含宫格/网格描述 → 输出类型 1（宫格/网格图片）

步骤4：检测差异化关键词
  └─ 如果包含差异化需求 → 输出类型 2（差异化多图）

步骤5：其他情况
  └─ 输出类型 4（常规生图）

⚠️ 重要提示：
- 用户上传图片必须优先判断，无论用户描述如何，都应识别为类型3（图生图）
- 例如：用户上传图片并说"生成一张图片" → 类型3，而不是类型4
```

## 类型 1：宫格/网格图片

### 定义

用户需要生成多张图片，并按照特定的宫格或网格布局合成一张图片。

### 触发条件

- 条件一：生成几宫格图片，例如"三宫格"、"四宫格"、"九宫格"
- 条件二：生成几x几网格，例如"3x3网格"、"4x4网格"、"2x2网格"

### 处理特点

- 生成单个提示词字符串，包含完整的宫格布局描述
- 保留每张图片的尺寸比例信息
- 不进行差异化拆分，保持整体性

### 示例

**输入**：
```
生成一张四宫格图片，展示四季变化，每个格子代表一个季节
```

**输出**：
```json
["四宫格图片，左上格展示春季，绿意盎然，鲜花盛开；右上格展示夏季，阳光明媚，蓝天白云；左下格展示秋季，金黄落叶，秋高气爽；右下格展示冬季，白雪皑皑，银装素裹。每个格子保持统一风格，整体画面和谐协调"]
```

### 注意事项

- 宫格布局信息会被完整保留在提示词中
- 每个格子的内容描述会按照顺序组织
- 输出是单个字符串，不是数组

---

## 类型 2：差异化多图

### 定义

用户需要生成多张具有明显差异的图片，这些差异体现在视角、角度、场景、风格等方面。

### 触发条件

- 要求不同的视角、角度、场景、风格
- 要求多样化的展示方式
- 明确提到需要"不同"、"多种"、"各种"等体现差异的词汇
- 要求生成多张图片且每张有不同特点

### 处理特点

- 用户未描述差异化信息时，自动补充具体的差异化描述（视角、角度、场景、风格等）
- 确保每个提示词的差异化描述都是不同的
- 输出为数组格式，数组长度等于图片数量

### 示例

**输入**：
```
生成三张不同风格的赛博朋克城市夜景
```

**输出**：
```json
[
  "赛博朋克风格的城市夜景，霓虹灯光闪烁，未来感十足，蓝色调为主，高楼大厦林立，街道空旷",
  "赛博朋克风格的城市夜景，俯视角度，从高空俯瞰城市，紫红色调为主，霓虹灯光形成美丽的光带",
  "赛博朋克风格的城市夜景，街道视角，雨后路面反射霓虹灯光，绿色调为主，行人稀少，氛围神秘"
]
```

### 自动补充的差异化维度

- **视角**：平视、俯视、仰视、鸟瞰、街道视角等
- **角度**：正面、侧面、斜角、远景、近景等
- **场景**：室内、室外，白天、夜晚、特定地点等
- **风格**：色调，光线、氛围、艺术风格等

### 注意事项

- 用户如果有描述差异化信息，直接用用户的描述信息，不得随意更改
- 用户没有描述差异化信息时，补充具体的差异化描述
- 每个提示词的差异化维度会尽量不同

## 类型 3：图生图

### 定义

基于参考图片进行图像生成、编辑、融合、风格迁移等操作，需要提供原始参考图。

### 触发条件

- 用户描述包含编辑或融合操作关键词
- 编辑关键词："切割"、"分割"、"裁剪"、"修改"、"处理"、"编辑"、"去除"、"删除"、"改"
- 融合关键词："融合"、"结合"、"合成"、"混合"、"叠加"
- 用户提及"基于生成的图片"、"修改这张图"、"融合这两张图"等指代性语言
- 用户上传了图片文件

### 处理特点

- 任务优先级最高，识别到关键词或用户上传图片直接输出类型 3
- 输出为数组格式

### 常见场景

1. **图片编辑**：对已有图片进行修改、裁剪、删除元素等操作
2. **图片融合**：将多张图片进行融合、合成、叠加
3. **风格迁移**：基于参考图进行风格转换
4. **图生图**：基于参考图生成新的图片

### 参考图来源

图生图的参考图可能来自以下两种情况：

1. **上一轮生成的图片**：
   - 从对话上下文中获取上一轮MCP工具返回的图片URL
   - URL格式：`https://mdn.alipayobjects.com/doraemon_plugin/afts/img/xxx/original`
   - 用户通常会说："基于生成的图片"、"修改刚才的图片"等

2. **用户上传的图片**：
   - 用户在当前对话中上传了图片文件
   - 图片URL会包含在用户消息的附件信息中
   - 用户通常会说："修改这张图片"、"把这个图片裁剪一下"等

### 参考图获取逻辑

```
步骤1：识别是否为图生图需求
  ├─ 检测关键词：修改、改、融合、合成、裁剪、去掉、添加等
  └─ 检测指代词：上面的图片、刚才的图片、这张图、这两张图等
  └─ 检查用户是否上传了图片

步骤2：确定参考图来源（按优先级）
  ├─ 优先级1：检查用户是否上传了图片
  │  └─ 如果用户消息中包含图片附件 → 提取图片URL
  ├─ 优先级2：从对话上下文获取上一轮生成的图片URL
  │  ├─ 查找上一轮对话中MCP工具返回的图片URL
  │  └─ URL格式示例：https://mdn.alipayobjects.com/doraemon_plugin/afts/img/xxx/original
  └─ 如果都没有找到 → 提示用户提供参考图

步骤3：构建MCP调用
  └─ 调用时必须包含referenceImageUrls参数
```

### 示例

**示例1：图片编辑 - 基于上一轮生成的图片**
```
用户输入：基于上面生成的图片，新春两个字改成春节，其他不变
```

**示例2：图片编辑 - 基于用户上传的图片**
```
用户输入：[上传了一张图片] 将这张图片裁剪成16:9的比例
用户输入：[上传了一张图片] 去掉这张图片中的英文文字
```

**示例3：图片融合 - 融合两张图片**
```
用户输入：[上传了两张图片] 将这两张图片融合在一起，左边是风景照，右边是人像照，自然过渡
用户输入：把这两张图片合成一张，上半部分是风景，下半部分是城市
```

**示例4：直接编辑请求（需从上下文或用户上传获取图片）**
```
用户输入：将这张图片裁剪成16:9的比例
用户输入：裁剪图片，比例16:9
```

**输出示例（图片编辑）**：
```json
["图片中把新春两个字改成春节，其他元素保持不变"]
```

**输出示例（图片融合）**：
```json
["将两张图片融合，左边是风景照，右边是人像照，中间自然过渡，整体画面和谐统一"]
```

### 常见操作类型

**图片编辑类**：
- **切割/分割**：将图片切割成多个部分
- **裁剪**：调整图片的尺寸或比例
- **修改**：对图片内容进行局部修改（文字、颜色、元素等）
- **去除/删除**：移除图片中的某些元素
- **处理**：对图片进行整体处理（滤镜，调色等）

**图片融合类**：
- **融合**：将多张图片融合成一张，自然过渡
- **合成**：将多张图片按特定布局合成
- **叠加**：将一张图片叠加到另一张图片上
- **混合**：将多张图片的元素混合在一起

### 注意事项

- ⚠️ **重要**：图生图操作**必须**提供参考图片 URL
- 参考图来源优先级：
  1. **优先检查用户是否上传了图片**：从用户消息的附件信息中提取图片URL
  2. **其次从对话上下文获取**：查找上一轮MCP工具返回的图片URL
- 如果用户未上传图片且上下文中也没有上一轮生成的图片，应提示用户提供参考图
- 描述要具体说明操作的内容和方式（编辑或融合）
- 处理后的图片保持原有风格和特征（编辑场景）或自然融合（融合场景）
- 在多轮对话场景中，需要追踪和保存每一轮生成的图片URL

## 类型 4：常规生图

### 定义

标准的单张或多张图片生成需求，不涉及宫格布局、差异化要求或参考图片。

### 触发条件

- **必须条件**：用户**未上传图片**
- 不满足类型 1、2、3 的所有常规需求
- 简单的图片生成请求
- 需要生成多张相同或相似主题的图片

### ⚠️ 重要排除

- 如果用户上传了图片 → **必须是类型3（图生图）**，不能是类型4
- 如果包含图生图关键词 → **必须是类型3（图生图）**，不能是类型4

### 常见场景

1. **单张图片生成**：生成一张简单的图片
2. **多张相似图片**：生成多张相同或相似主题的图片
3. **无特殊需求**：不涉及宫格、差异化或图生图的常规需求

### 注意事项

- 必须确认用户**未上传图片**，否则应为类型3
- 不涉及参考图片，不需要提供referenceImageUrls参数
- 提示词数组中的每个元素都包含相同的主题描述

### 类型 1 输出示例（宫格/网格）

```json
["三宫格图片，左侧展示清晨的森林，中间展示正午的森林，右侧展示黄昏的森林。每个格子统一风格，保持视觉连贯性"]
```

### 类型 2 输出示例（差异化多图）

```json
[
  "赛博朋克风格的城市夜景，霓虹灯光闪烁，未来感十足，蓝色调为主",
  "赛博朋克风格的城市夜景，俯视角度，高楼林立，紫红色调为主",
  "赛博朋克风格的城市夜景，街道视角，雨后反射，绿色调为主"
]
```

### 类型 3 输出示例（图生图）

```json
["将图片中的背景替换为蓝天白云"]
```

或

```json
["将两张图片融合，左边是风景照，右边是人像照，自然过渡"]
```

### 类型 4 输出示例（常规生图）

```json
[
  "一只可爱的猫咪在阳光下睡觉",
  "一只可爱的猫咪在阳光下睡觉",
  "一只可爱的猫咪在阳光下睡觉"
]
```

## 关键限制

- 仅根据需求描述输出，不添加个人判断或假设
- 输出必须是数组格式，不包含解释性内容
- 禁止使用 Markdown 代码块包裹
- 专有名词必须原样保留，不得省略或替换
